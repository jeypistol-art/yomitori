# YOMITORI DocuTask MVP実装タスク分解

## 1. 目的

YOMITORI DocuTask のMVPを、実装可能な単位に分解する。

MVPのゴールは、管理会社向けに以下の一連の流れを動かすこと。

1. ユーザーがログインする
2. 組織を作る
3. 書類を登録する
4. PDF / 画像 / テキストを保存する
5. OCR / AI抽出を行う
6. 左右2ペインの承認画面で確認する
7. 承認してタスク・リマインド・台帳へ反映する
8. 月次未処理一覧で対応漏れを確認する
9. 使用件数と追加パックを管理する

## 2. 実装前提

### 技術構成

- Cloudflare Workers / Pages
- Cloudflare R2
- Neon PostgreSQL
- Stripe
- OpenAI API
- Google Auth
- Google Calendar API または .ics

### 初期対象

不動産・施設などの管理会社。

### 初期対応書類

- 行政・自治体からの通知
- 契約更新案内
- リース契約更新案内
- 保険満期案内
- テナント契約更新案内
- 法改正に伴う提出物案内

### 実装方針

- 既存TENsNAPの技術資産を流用できる部分は流用する
- DBは作成済みDDLを基準にする
- まずはMVPに必要な最短導線を優先する
- AI抽出結果は自動確定しない
- 承認時に正式データへ反映する
- 高度な権限、複雑な承認フロー、差分比較は後続

## 16. 開発順の推奨

最短で価値検証するなら、この順番にする。

1. 既存TENsNAP調査
2. プロジェクト基盤
3. DB接続
4. Googleログイン
5. 書類登録
6. R2保存
7. テキスト貼り付けからAI抽出
8. 承認画面
9. タスク作成
10. 台帳・未処理一覧
11. PDF / 画像OCR
12. 使用件数
13. Stripe

理由:

- OCRより先に、テキスト入力でAI抽出と承認画面を固められる
- 承認画面が固まると、DB・AI・タスクの接続が見える
- PDF / 画像OCRはTENsNAP流用調査後に入れた方が効率が良い
- Stripeは使用件数ロジックが固まってから入れる方が安全

## 17. 主要リスクと対策

### OCR・AI処理が重い

対策:

- 初期はテキスト貼り付け導線も必ず用意する
- PDF / 画像処理はジョブ化する
- 処理中ステータスを明確に出す

### AI抽出が安定しない

対策:

- JSON Schemaで検証する
- raw_outputとnormalized_outputを保存する
- テスト書類セットを作る
- confidenceとwarningsを承認画面に出す

### 課金単位が分かりにくい

対策:

- 月間処理件数に統一する
- 追加パックも件数に統一する
- 極端にページ数の多い書類は規約で制限する

### 管理会社向けの価値がぼやける

対策:

- 管理対象と取引先をMVPから入れる
- 行政通知と契約更新案内を優先する
- 月次未処理一覧を必ず入れる
